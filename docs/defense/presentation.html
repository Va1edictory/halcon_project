项目背景

✕ 人工目检的痛点

  • 🔴 效率低 — 依赖人工逐件检查
  • 🔴 主观性强 — 不同质检员标准不一
  • 🔴 易疲劳 — 长时间工作漏检率上升
  • 🔴 难追溯 — 无法系统记录缺陷数据

✓ 自动检测的优势

  • 🟢 7×24 无人值守运行
  • 🟢 基于模型的标准化判定
  • 🟢 检测一致性高，不受疲劳影响
  • 🟢 结果数字化，便于统计与追溯
核心目标：利用 HALCON DLT 深度学习平台，构建一套能自动判断晶体管 OK/NG 并标注缺陷位置的非接触式检测系统。

技术流水线详解

① 图像预处理

步骤算子变换原因
通道转换compose31ch → 3ch模型要求 3 通道输入
尺寸缩放zoom_image_size900² → 256²模型输入固定 256×256
类型转换convert_image_typebyte → real模型需要浮点类型
值域映射scale_image[0,255] → [-127,128]匹配训练时的值域
域的设置full_domain—避免 domain 裁剪
样本生成gen_dl_samples_from_images→ DLSample避免 iconic/control 冲突

② 异常推理

apply_dl_model → DLResult
get_dict_object → AnomalyImage（用 object 接口避免类型冲突）
get_dict_tuple → AnomalyScore
Score > Threshold ? NG : OK

③ 缺陷定位

scale_image_max
值域拉伸
threshold
二值化
connection
分离连通域
select_shape
过滤噪点
shape_trans
生成矩形框

④ 坐标映射

问题：热力图 256×256，原图 900×900，直接叠加偏移严重
解决：hom_mat2d_scale + affine_trans_region 缩放至原图尺寸

关键技术难点与解决方案

难点问题描述解决方案
HDevelop 26 XML 适配 HALCON 26 使用 <l>/<c> 元素存储代码，与旧版格式不兼容 分析 HDevelop 原生导出格式，按规范编写 XML
Iconic/Control 类型冲突 set_dict_tuple 不接收 iconic 图像变量 改用 gen_dl_samples_from_images
异常图提取失败 get_dict_tuple 返回 control 值 改用 get_dict_object
preprocess_dl_model 缺失 HALCON 26 已移除此算子 手动实现 5 步预处理流水线
坐标偏差 256² vs 900² 尺寸不匹配 仿射变换 affine_trans_region
文本不可见 disp_message 坐标系错误 'window' → 'image' 坐标系
通道不匹配 原图 3ch vs 热力图 1ch rgb1_to_gray 统一为单通道
批量变量未定义 ImgWidth/ImgHeight 批量循环中未初始化 每张图片单独 get_image_size
